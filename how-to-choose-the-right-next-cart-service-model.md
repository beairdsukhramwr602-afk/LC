# How to Choose the Right Next-Cart Service Model

Choosing the right Next-Cart service model is a decision about execution responsibility, migration complexity, internal capacity, and whether the project needs customization or modification work. Price matters, but the safest service choice starts with what the migration actually requires.

Next-Cart offers three service models: Standard Service, Managed Service, and Custom Service. Standard Service is customer-led execution. Managed Service is Next-Cart-led execution. Custom Service is required when the project needs customization, modification, tailored Add-ons, custom Add-ons, or broader bespoke handling.

This article helps customers decide which model fits their migration project.

### Start With Two Core Questions <a href="#start-with-two-core-questions" id="start-with-two-core-questions"></a>

A service model decision becomes clearer when the customer separates two questions:

1. Who should perform the migration process?
2. Does the project require customization or modification work?

The first question separates Standard Service from Managed Service.

The second question determines whether Custom Service is required.

This distinction matters because a project can need customization without needing Next-Cart to perform the full migration process. It can also need Next-Cart-led execution without requiring customization.

### When Standard Service Is the Better Fit <a href="#when-standard-service-is-the-better-fit" id="when-standard-service-is-the-better-fit"></a>

Standard Service is usually a strong fit when the customer can perform the migration process and the project fits default migration-tool capability.

It works best when:

* the customer is comfortable operating the migration tool
* the migration path looks predictable after Demo Migration
* the customer can review and validate the result internally
* required Add-ons fit within the standard default capability
* no Add-on modification or custom Add-on is needed
* no Custom Cart, custom field, third-party data, or bespoke handling requirement is involved
* the customer wants direct control over timing and execution

Standard Service is not the same as unsupported self-service. The customer still receives access to expert support. The difference is that the customer remains responsible for operating the migration process.

### When Standard Service May Not Be Enough <a href="#when-standard-service-may-not-be-enough" id="when-standard-service-may-not-be-enough"></a>

Standard Service may not be the right fit if the project requires more than customer-led execution through default capability.

Warning signs include:

* the customer does not have enough time or confidence to operate the migration
* Demo Migration results are hard to interpret without expert help
* internal teams cannot validate the target result confidently
* filtering, mapping, or data configuration needs exceed the default Add-on capability
* the project includes Custom Cart handling
* important data depends on third-party apps, plugins, modules, or extensions
* custom fields or outside-system identifiers must be preserved in a specific way
* migration-tool behavior needs modification

Some of these issues point toward Managed Service. Others require Custom Service. The difference depends on whether the problem is execution burden or customization need.

### When Managed Service Is the Better Fit <a href="#when-managed-service-is-the-better-fit" id="when-managed-service-is-the-better-fit"></a>

Managed Service is usually a strong fit when the migration can be handled through the default migration tool capability, but the customer wants Next-Cart to perform the migration process.

It works best when:

* the customer wants to reduce internal execution work
* the migration appears feasible through the standard configuration
* required Add-ons can be used within the default capability
* the customer wants Next-Cart experts to configure and run the migration
* the customer can still review and validate the result
* the project does not require customization or modification work

Managed Service is especially useful when the customer has a clear migration goal but does not want to manage the operational workload of running the migration.

### When Managed Service May Not Be Enough <a href="#when-managed-service-may-not-be-enough" id="when-managed-service-may-not-be-enough"></a>

Managed Service does not automatically include custom work.

A project moves beyond Managed Service when it requires customization or modification, such as:

* modifying a standard Add-on beyond its default capability
* creating a custom Add-on
* adjusting migration-tool behavior
* handling Custom Cart as Source Cart or Target Cart
* transforming data through bespoke rules
* preserving custom fields or outside-system identifiers beyond default support
* handling app, plugin, module, or extension data outside the standard platform model

When any customization or modification work is required, the project is handled through Custom Service.

### When Custom Service Is Required <a href="#when-custom-service-is-required" id="when-custom-service-is-required"></a>

Custom Service is required whenever the project needs work beyond the default migration tool capability or the standard Add-on capability.

This includes:

* tailored Add-ons
* custom Add-ons
* migration-tool modification
* Custom Cart handling
* third-party app, plugin, module, or extension data
* custom fields
* outside-system identifiers
* platform capability limitations that need tailored handling
* special transformation or data-configuration rules
* broader bespoke migration requirements

Custom Service should be chosen because the project needs custom handling, not only because the project feels important or large. A large but standard migration may still fit Standard Service or Managed Service if it does not require customization.

### Custom Service Does Not Always Mean Managed Execution <a href="#custom-service-does-not-always-mean-managed-execution" id="custom-service-does-not-always-mean-managed-execution"></a>

Custom Service and Managed Service answer different questions.

Managed Service answers:

> Who performs the migration process?

Custom Service answers:

> Does the migration require customization or modification work?

A customer can purchase Custom Service for customization work and still perform the migration process themselves if migration management is not included in the final plan. The customer can also include optional migration management if they want Next-Cart to perform the migration process with the custom work in place.

This is important for customers who need a tailored Add-on or custom Add-on but still want to manage execution internally.

### How Demo Migration Supports Service Choice <a href="#how-demo-migration-supports-service-choice" id="how-demo-migration-supports-service-choice"></a>

Demo Migration can provide early evidence for the service decision.

A clear, predictable demo result may support Standard Service when the customer is comfortable performing the migration.

A clean but workload-heavy demo result may support Managed Service when the customer wants Next-Cart to handle execution.

A demo result that reveals custom mapping, custom fields, app-owned data, Custom Cart issues, or Add-on limitations points toward Custom Service.

The demo should not be judged only by whether records appear. It should be judged by whether the sample result preserves the business meaning that must survive after launch.

### How Entity Points Affect Service Choice <a href="#how-entity-points-affect-service-choice" id="how-entity-points-affect-service-choice"></a>

Entity Points help estimate capacity, but they do not decide the service model by themselves.

A project with high Entity Points may still fit Standard Service if the customer can perform the migration and the project does not require customization.

A project with lower Entity Points may still require Custom Service if the data structure is unusual, the target platform needs bespoke handling, or the customer needs a tailored Add-on or custom Add-on.

Entity Points answer the capacity question. Service choice answers the execution and customization question.

### How Add-ons Affect Service Choice <a href="#how-add-ons-affect-service-choice" id="how-add-ons-affect-service-choice"></a>

Standard Add-ons can be used with Standard Service, Managed Service, and Custom Service.

Using a standard Add-on within the default capability does not automatically make the project Custom.

For example:

* a Standard Service customer can use the Data Filter Add-on and still perform the migration
* a Managed Service customer can use Advanced Data Mapping within the default capability and have Next-Cart execute the migration
* a Custom Service project can include standard, tailored, or custom Add-ons as part of a broader custom plan

The service implication changes when the Add-on needs modification beyond the default capability or when the customer requests a custom Add-on. That work is handled through Custom Service.

### How Custom Cart Affects Service Choice <a href="#how-custom-cart-affects-service-choice" id="how-custom-cart-affects-service-choice"></a>

Any migration involving Custom Cart as Source Cart or Target Cart requires Custom Service.

A Custom Cart is not simply a platform with a different connection method. The main issue is that Custom Cart projects often require interpretation, transformation, validation sensitivity, and migration-tool fine-tuning beyond the standard supported-cart model.

If Custom Cart is involved, the service decision always starts with Custom Service.

### How Internal Capacity Affects Service Choice <a href="#how-internal-capacity-affects-service-choice" id="how-internal-capacity-affects-service-choice"></a>

Service choice is also about the customer’s internal capacity.

Customers should consider whether they can:

* operate the migration process confidently
* prepare access and source-store information
* interpret Demo Migration results
* make configuration decisions
* review migrated records
* validate business-critical workflows
* coordinate launch timing
* decide whether Add-ons or Custom Service are needed

If the customer has limited time or confidence to manage these responsibilities, Managed Service may be more appropriate even when the migration does not require customization.

If the customer can operate the process but needs customized logic or modified Add-ons, Custom Service may be needed without necessarily adding migration management.

### Practical Service-Fit Scenarios <a href="#practical-service-fit-scenarios" id="practical-service-fit-scenarios"></a>

#### Scenario 1: Predictable migration, capable internal team <a href="#scenario-1-predictable-migration-capable-internal-team" id="scenario-1-predictable-migration-capable-internal-team"></a>

The customer understands the migration process, can operate the tool, and the demo result looks predictable. Required Add-ons fit default capability.

**Likely fit:** Standard Service.

#### Scenario 2: Predictable migration, limited execution capacity <a href="#scenario-2-predictable-migration-limited-execution-capacity" id="scenario-2-predictable-migration-limited-execution-capacity"></a>

The migration appears feasible through default capability, but the customer wants Next-Cart to configure and perform the migration process.

**Likely fit:** Managed Service.

#### Scenario 3: Standard Add-on is enough <a href="#scenario-3-standard-add-on-is-enough" id="scenario-3-standard-add-on-is-enough"></a>

The customer needs to filter selected data or adjust mapping through available Add-on settings, but the Add-on does not need modification.

**Likely fit:** Standard Service or Managed Service, depending on who should perform execution.

#### Scenario 4: Add-on needs modification <a href="#scenario-4-add-on-needs-modification" id="scenario-4-add-on-needs-modification"></a>

The customer needs a filtering, mapping, or data-configuration result that exceeds default Add-on capability.

**Required fit:** Custom Service.

#### Scenario 5: Custom Cart is involved <a href="#scenario-5-custom-cart-is-involved" id="scenario-5-custom-cart-is-involved"></a>

The source or target platform is Custom Cart and requires interpretation beyond the supported-cart model.

**Required fit:** Custom Service.

#### Scenario 6: Customer needs custom work but wants to self-run <a href="#scenario-6-customer-needs-custom-work-but-wants-to-self-run" id="scenario-6-customer-needs-custom-work-but-wants-to-self-run"></a>

The customer needs a tailored Add-on or custom Add-on, but still wants to operate the migration process internally.

**Required fit:** Custom Service for the customization work, without optional migration management unless the final plan includes it.

#### Scenario 7: Customer needs custom work and wants Next-Cart execution <a href="#scenario-7-customer-needs-custom-work-and-wants-next-cart-execution" id="scenario-7-customer-needs-custom-work-and-wants-next-cart-execution"></a>

The customer needs customization and also wants Next-Cart to perform the migration.

**Required fit:** Custom Service with migration management included in the final plan.

### Questions to Ask Before Choosing <a href="#questions-to-ask-before-choosing" id="questions-to-ask-before-choosing"></a>

Before selecting a service model, customers should ask:

* Do we want to perform the migration ourselves?
* Do we have enough internal time and confidence to operate the migration process?
* Did Demo Migration show predictable results?
* Do we need Add-ons for filtering, mapping, or data configuration?
* Can the required Add-ons work within default capability?
* Does any Add-on need modification?
* Do we need a custom Add-on?
* Is Custom Cart involved?
* Do third-party apps, plugins, modules, or extensions affect the data?
* Are custom fields or outside-system identifiers important after launch?
* Do we need Next-Cart to manage execution, customization, or both?

These questions help separate service preference from service requirement.

### Conclusion <a href="#conclusion" id="conclusion"></a>

The right Next-Cart service model depends on two decisions: who should perform the migration and whether the project requires customization or modification work. Standard Service fits customer-led execution through default capability. Managed Service fits Next-Cart-led execution through default capability. Custom Service is required when the project needs tailored Add-ons, custom Add-ons, Custom Cart handling, migration-tool adjustment, or broader bespoke handling.

Service choice should be based on evidence from Demo Migration, the customer’s internal capacity, Add-on needs, Entity Points planning, and any custom requirements that affect the expected target result.

Use the service model decision as a planning checkpoint, not only a purchase step. If your demo results, Add-on needs, Custom Cart involvement, custom fields, third-party data, or internal execution capacity are unclear, Live Chat can help clarify whether Standard Service, Managed Service, or Custom Service is the safer path.

### FAQs <a href="#faqs" id="faqs"></a>

#### How do I choose between Standard Service and Managed Service? <a href="#how-do-i-choose-between-standard-service-and-managed-service" id="how-do-i-choose-between-standard-service-and-managed-service"></a>

Choose Standard Service when you want to perform the migration yourself. Choose Managed Service when you want Next-Cart to perform the migration process using default capability and purchased standard Add-ons.

#### When is Custom Service required? <a href="#when-is-custom-service-required" id="when-is-custom-service-required"></a>

Custom Service is required whenever the project needs customization or modification work, including tailored Add-ons, custom Add-ons, migration-tool adjustment, Custom Cart handling, custom fields, third-party data, or other bespoke requirements.

#### Does high Entity Points volume automatically mean I need Managed Service or Custom Service? <a href="#does-high-entity-points-volume-automatically-mean-i-need-managed-service-or-custom-service" id="does-high-entity-points-volume-automatically-mean-i-need-managed-service-or-custom-service"></a>

No. Entity Points measure capacity, not service fit by themselves. A high-volume migration can still fit Standard Service if it is predictable and the customer can perform the migration.

#### Can a smaller migration still require Custom Service? <a href="#can-a-smaller-migration-still-require-custom-service" id="can-a-smaller-migration-still-require-custom-service"></a>

Yes. A smaller migration can require Custom Service if it involves custom fields, Custom Cart, modified Add-ons, custom Add-ons, or bespoke migration behavior.

#### Does using a standard Add-on make the service Custom? <a href="#does-using-a-standard-add-on-make-the-service-custom" id="does-using-a-standard-add-on-make-the-service-custom"></a>

No. Using a standard Add-on within default capability does not automatically make the service Custom. Add-on modification or a custom Add-on request is handled through Custom Service.

#### Can I choose Custom Service but still perform the migration myself? <a href="#can-i-choose-custom-service-but-still-perform-the-migration-myself" id="can-i-choose-custom-service-but-still-perform-the-migration-myself"></a>

Yes. Custom Service means customization or modification work is required. The customer may still perform the migration unless migration management is included in the final plan.

#### Should I choose the service model before Demo Migration? <a href="#should-i-choose-the-service-model-before-demo-migration" id="should-i-choose-the-service-model-before-demo-migration"></a>

You may have an initial expectation before Demo Migration, but the demo result often gives better evidence for confirming whether Standard Service, Managed Service, or Custom Service is the right fit.

#### What if I am unsure which service model fits? <a href="#what-if-i-am-unsure-which-service-model-fits" id="what-if-i-am-unsure-which-service-model-fits"></a>

If the right fit is unclear, review the Demo Migration result, Add-on needs, internal execution capacity, Custom Cart involvement, and any custom data requirements. Live Chat can help clarify the safest service path.
